# Document Polishing

Refine an existing document draft so it is easier to read, review, and submit while preserving its intended meaning.

This skill is an editor and refiner, not a primary document author.

If the main problem is missing substance rather than poor expression, another document-authoring skill should usually run first.

Its job is to improve:

- clarity;
- structure;
- consistency;
- readability;
- scannability;
- submission readiness.

Its job is not to invent missing product or technical decisions just to make a document look complete.

## Use this skill when

Use this skill when the user already has a draft, rough document, or substantial document content and wants help with any of the following:

- tightening wording;
- reorganizing sections;
- reducing repetition;
- improving readability;
- unifying tone;
- making a document look more review-ready or submission-ready;
- cleaning up a draft produced by another document-writing skill.

Typical requests may sound like:

- “Polish this spec.”
- “Make this technical doc cleaner.”
- “Tighten this draft without changing meaning.”
- “Reorganize this document so it reads better.”
- “Turn this rough doc into something more presentable.”

## Do not use this skill when

Do not use this skill when the main need is:

- creating a document mostly from scratch;
- deciding feature scope, non-goals, or acceptance direction;
- reconstructing technical flow from code or system behavior;
- filling major content gaps that require new product or technical decisions;
- pretending unresolved ambiguity is already resolved.

In those cases, an authoring or clarification skill should own the work first.

## Expected inputs

An existing draft or substantial document content is normally required.

Common inputs include:

- a feature spec draft;
- a technical document draft;
- rough prose or partially structured notes;
- a merged draft from earlier conversations;
- a document generated by another skill that now needs cleanup.

Helpful optional inputs include:

- document type;
- target audience;
- desired polish level;
- specific editing priorities such as brevity, tone consistency, or section reorganization.

## Expected outputs

The default output should include:

1. a polished draft;
2. preserved unresolved questions or uncertainties where they still matter;
3. optionally, a short summary of notable edits or flagged ambiguities when useful.

The result should be clearer and more presentable without pretending to know more than the source material actually establishes.

If a draft is too skeletal to polish honestly, say so and identify what kind of authoring or clarification work is still missing.

## Editing posture

### 1. Preserve intent first

Before changing wording or structure, identify:

- the document's likely purpose;
- the target audience;
- what claims appear confirmed;
- what remains ambiguous or provisional.

Preserve the original intent unless the user explicitly asks for meaning changes.

### 2. Improve expression, not substance

Prefer improvements such as:

- rewriting awkward sentences;
- removing duplication;
- improving section order;
- tightening headings;
- making lists and comparisons easier to scan;
- standardizing tone and phrasing.

Do not silently add new scope, rationale, flow detail, or decisions just because the draft feels incomplete.

When a better document would require new content rather than better editing, flag that gap explicitly.

### 3. Surface gaps instead of faking certainty

If the draft has holes, contradictions, or unclear ownership:

- preserve the uncertainty when it still matters;
- flag ambiguity when needed;
- avoid rewriting open questions into settled statements.

### 4. Preserve audience fit

Polish the document for its intended audience.

Examples:

- a feature-facing document should stay readable for TD / PM / QA;
- a technical document should stay useful for engineers and maintainers.

Do not flatten the document into generic prose that loses its audience fit.

## Polish modes

### Light polish

Use for sentence cleanup, tone consistency, and minor readability improvements without major restructuring.

### Structural polish

Use for reorganizing sections, merging repeated content, and improving information flow while preserving meaning.

### Submission polish

Use for turning a solid draft into a more presentation-ready version with stronger consistency, scannability, and finish.

Even in submission polish mode, do not invent missing decisions or erase unresolved issues.

## What to preserve

- underlying intent;
- important caveats;
- known issues;
- open questions;
- audience-appropriate framing;
- meaningful distinctions already present in the draft.

## What to improve

- clarity;
- structure;
- readability;
- redundancy;
- tone consistency;
- heading quality;
- document flow;
- submission readiness.

## Constraints and anti-patterns

- Do not use this as a substitute for writing the document in the first place.
- Do not silently resolve unknowns to make the text sound smoother.
- Do not convert assumptions into facts.
- Do not remove caveats that materially change meaning.
- Do not treat a sparse outline as if it were already a polishable full draft when major authoring is still missing.
- Do not blur the difference between feature-facing and engineering-facing documents.
- Do not over-normalize the prose so hard that the document loses useful nuance.
- Do not impose project-specific storage or formatting conventions by default.

## Good result characteristics

A good result from this skill should:

- read more cleanly than the input draft;
- preserve the source document's meaning and audience fit;
- expose unresolved ambiguity rather than burying it;
- reduce friction for review or submission;
- remain clearly derived from an existing draft instead of pretending to be original authoring.
